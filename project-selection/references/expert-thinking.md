# How an Expert Thinks About Project Selection

A beginner picks projects by asking *"which of these is most impressive?"*

An expert never asks that. An expert asks: *"Which 3 projects make the hiring manager believe this candidate has already solved this exact problem?"*

That shift changes everything:

- **The candidate's favorite project is often the wrong pick.** Engineers are proud of the hardest or flashiest project. None of that matters if it doesn't speak to *this* JD. Pride is not relevance.
- **A "boring" project frequently wins.** A modest, well-quantified CRUD service that lands squarely on a `priority: A` must-have beats a flashy ML side project the role never asks for.
- **You are casting a panel, not ranking soloists.** The 3 projects are a team. The question is never "what are the 3 best projects" — it's "what 3 projects, *together*, cover the most ground." The third pick is often chosen *because* it covers what the first two missed, even if it ranks 6th alone.
- **What you leave out is a decision too.** Selection is editing. Editing is mostly cutting.

The score (Step 3) keeps the agent honest and consistent. The judgment (Step 4) keeps it from being fooled by a number. Hold both.
