# CAR Method For Portfolio Points

> **Location:** `/opt/data/profiles/<PROFILE_SLUG>/skills/point-creation/references/CAR.md`
> **Referenced by:** `point-creation` skill (main guide) → loaded when the agent needs to draft a CAR-format bullet (problem-solving story).

This file explains how to use the `CAR` method to build stronger portfolio points for `work experience` and `projects`.

## What CAR Means

`CAR` stands for:
1. `Challenge`
2. `Action`
3. `Result`

In plain language:
- `Challenge`: the business, system, workflow, or operational problem
- `Action`: what you did to address it
- `Result`: what improved because of your work

CAR is close to STAR, but more compressed. It removes explicit task framing and stays focused on: the problem, the intervention, the payoff.

## Core Rule

Use CAR when the point is really about `solving something`.

CAR often starts from the difficulty or pressure point. That makes CAR strong when:
- the problem is important to understand first
- the action only makes sense with challenge context
- the value of the work comes from how you resolved the issue

For this portfolio, the final point should usually:
- make the challenge clear without overexplaining it
- keep the action concrete
- end with the payoff, impact, or measurable result
- stay compressed instead of reading like a long before/after story

In other words: think in `problem → intervention → outcome`, write in `compressed CAR`.

## How To Use CAR To Create A Point

### Step 1: Write the raw CAR notes
- `Challenge:`
- `Action:`
- `Result:`

Keep them factual.

### Step 2: Make sure the challenge is real
The challenge should require: judgment, prioritization, design decisions, technical tradeoffs, or process intervention.

Good challenge types: rising load/latency, unreliable workflows, manual coordination overhead, security risk, missing observability, slow delivery.

Weak challenge types: generic daily duties, obvious responsibilities with no tension, vague improvement claims.

### Step 3: Keep the action brief but concrete
The action should explain the intervention, not narrate every implementation detail.

### Step 4: Frontload the result when possible
Recruiters scan quickly — the payoff should not be buried. Even if you think in CAR, the final sentence can be reordered so the result lands first.

### Step 5: Check if the point is portfolio-grade
- Is the challenge understandable? Is the action specific? Is the result visible? Does it show problem-solving?

## When To Use CAR

Use CAR when the point describes a `challenge you solved`:
- performance bottlenecks
- process improvement
- reliability fixes
- workflow redesign
- security/access-control interventions
- cross-team alignment problems
- operational inefficiencies
- adoption blockers

## When CAR Is NOT The Best Fit

- broad scope summaries
- pure technology statements
- project overview sentences
- accomplishment where the metric matters more than the challenge
- recognition/selection points without problem-solving context

## Where To Use CAR In This Portfolio

1. **Role Overview Points** — 1–2 CAR-shaped points in a 4–5 point overview
2. **Detailed Workstream Card Points** — Heavy CAR usage; most cards support at least 1 strong CAR point
3. **Project Overview Points** — When the project exists to solve a clear pain point
4. **Deep Dive Project Sections** — Subsystems that resolved known issues, design decisions that removed friction

## Point Patterns That Work Well

### Pattern A: Challenge → Action → Result
`To address [challenge], [action], resulting in [result].`

### Pattern B: Result → Action → Challenge
`Achieved [result] by [action] in response to [challenge].`

### Pattern C: Challenge → Result → How
`[Challenge] was addressed by achieving [result] through [action].`

## Practical Rules For This Repo

1. Use CAR when the value comes from solving a concrete problem.
2. Keep the challenge short unless the constraint is the main story.
3. Keep the action visible, but do not let it dominate the sentence.
4. End with result, impact, or why the solution mattered.
5. Frontload the payoff if the sentence reads better that way.
6. If the metric is the main story, XYZ may be a better fit.
7. If the task and background matter more, STAR may be a better fit.

## If There Is No Metric

Use result types like: clearer access control, lower coordination overhead, easier debugging, stronger observability, better portability, improved workflow reliability, reduced manual work.

## Reusable Compression Prompts

- `What real problem existed here?`
- `Why did this issue matter?`
- `What exactly did I address it?`
- `What improved afterward?`
- `Can I show the payoff with a number, system change, or workflow improvement?`

## Final Principle

CAR is a way to show that the work had `a problem worth solving` and that you actually solved it. The best CAR points feel like: clear problem, concrete intervention, visible payoff.
