# STAR Method For Portfolio Points

> **Location:** `/opt/data/profiles/<PROFILE_SLUG>/skills/point-creation/references/STAR.md`
> **Referenced by:** `point-creation` skill (main guide) → loaded when the agent needs to draft a STAR-format bullet.

This file explains how to use the `STAR` method to build stronger portfolio points for `work experience` and `projects`.

The goal is not to copy interview-style STAR answers into the site.

The goal is to use STAR as a `drafting and thinking framework`, then compress it into points that are:
- specific
- outcome-aware
- grounded in real context
- easy to scan
- useful for portfolio writing, not just resumes

## What STAR Means

`STAR` stands for:

1. `Situation`
2. `Task`
3. `Action`
4. `Result`

In plain language:
- `Situation`: what context, challenge, system, or business condition existed
- `Task`: what you were responsible for solving, improving, building, or owning
- `Action`: what you actually did
- `Result`: what changed because of your work

## Core Rule

Use STAR to `discover` the point before you write it.

Do not dump all four sections mechanically into the final bullet.

For this portfolio, the final point should usually:
- keep `Situation` short
- often merge `Situation` and `Task`
- spend the most space on `Action`
- end with `Result`, impact, or why it mattered
- not force the four parts to appear as separate clauses in strict `S -> T -> A -> R` order

In other words:
- think in `STAR`
- write in `compressed STAR`

## How To Use STAR To Create A Point

### Step 1: Write the raw STAR notes

Before writing the final point, write four rough lines:
- `Situation:`
- `Task:`
- `Action:`
- `Result:`

Keep these rough and factual.

### Step 2: Decide what deserves emphasis

Not every STAR part gets equal weight. Ask:
- Is the context itself important?
- Is the ownership important?
- Is the implementation the most impressive part?
- Is the measurable result the strongest part?

For this portfolio, `Action + Result` usually carry the most value.

### Step 3: Compress the point

Turn the raw STAR notes into one point that reads naturally.

Preferred pattern:
`[Action]` + `so/for/because [Task or Situation]` + `which [Result or impact]`

### Step 4: Check if the point is portfolio-grade

A strong final point should pass this check:
- Can a reader understand the kind of work?
- Can a reader see what you actually did?
- Can a reader see why it mattered?
- Is the point specific enough to trust?
- If a metric exists, is it included?

## When To Use STAR For A Point

Use STAR when the point describes a `specific accomplishment, improvement, intervention, or ownership decision`.

STAR is especially strong for:
- performance optimization
- system redesign
- process improvement
- reliability or observability improvements
- cross-functional coordination with a concrete outcome
- architecture decisions tied to a clear business or technical problem
- feature delivery where the impact is known
- migration, automation, or security work with a before/after effect

## When STAR Is The Best Fit

STAR is the best fit for points like:
- `reduced load by 75%`
- `built a workflow that improved reliability`
- `created an access-control layer before LLM execution`
- `restructured a process that improved delivery or collaboration`
- `introduced observability that made debugging easier`

## When STAR Is Not The Best Fit

Do not force STAR into every point. STAR is weaker when the point is mostly:
- a broad scope summary
- a list of owned technologies
- a static role description
- a product overview sentence
- a long-running responsibility without a single clear outcome

## Where To Use STAR In This Portfolio

### 1. Role Overview Points — Use STAR selectively
- Best for: clear interventions with real numbers
- Mix: 1–3 STAR-shaped points in a 4–5 point overview

### 2. Detailed Workstream Card Points — Use STAR heavily
- Each card usually describes one major challenge, workstream, implementation direction, and outcome
- Most of the 3 points in a card can be drafted with STAR

### 3. Project Overview Points — Use STAR only when there's a clear problem-to-outcome story

### 4. Deep Dive Project Sections — Use STAR for subsystems, major features, technical challenges with visible payoff

## Point Patterns That Work Well

### Pattern A: Action → Context → Result
`Implemented [action] for [situation/task], which led to [result].`

### Pattern B: Situation/Task → Action → Result
`To address [situation/task], [action], resulting in [result].`

### Pattern C: Ownership → Action → Why It Mattered
`Owned [scope] and [action], creating/enabling [result].`

## Practical Rules For This Repo

1. Draft in STAR first if the work has a clear before/after story.
2. Keep the final point to one strong sentence whenever possible.
3. Merge `Situation` and `Task` unless both are independently important.
4. Keep `Action` visible.
5. End with `Result`, impact, or why the approach mattered.
6. Prefer real numbers over adjectives.
7. If no metric exists, use a concrete system outcome instead.

## Reusable Compression Prompts

- `What was the real problem here?`
- `Why did this work need to happen?`
- `What exactly did I change or build?`
- `What improved because of it?`
- `Can I prove the result with a number, system change, or workflow outcome?`

## Final Principle

STAR is not the final writing style. It is the method used to uncover the final writing.

For this portfolio, the best points usually feel like: clear action, grounded context, visible outcome.
