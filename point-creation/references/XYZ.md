# XYZ Method For Portfolio Points

> **Location:** `/opt/data/profiles/<PROFILE_SLUG>/skills/point-creation/references/XYZ.md`
> **Referenced by:** `point-creation` skill (main guide) → loaded when the agent needs to draft an XYZ-format bullet (result-first, metrics-forward).

This file explains how to use the `XYZ` method to build stronger portfolio points.

## What XYZ Means

`XYZ` stands for:
1. `X`: what you accomplished
2. `Y`: how the accomplishment was measured
3. `Z`: what you did to make it happen

The common formula is: `Accomplished [X] as measured by [Y], by doing [Z].`

## Core Rule

Use XYZ when you already know the result and want to make the point `sharper, shorter, and more credible`.

Unlike STAR, XYZ does not begin with situation or task. It begins with the accomplishment.

That makes it strong when:
- the result is the most persuasive part
- a metric exists
- the reader should notice the impact first

For this portfolio, the final point should usually:
- start with an active verb
- make `X` visible early
- include `Y` if it is real
- explain `Z` without turning the point into a tool list

In other words: think in `result → proof → method`, write in `compressed XYZ`.

## How To Use XYZ To Create A Point

### Step 1: Write the raw XYZ notes
- `X:` (accomplishment)
- `Y:` (measurement/proof)
- `Z:` (method)

### Step 2: Check whether Y is strong enough
Use percentages, counts, time reductions, scale numbers, rankings, or concrete system outcomes.

### Step 3: Write the first draft in formula form
`Accomplished [X] as measured by [Y], by doing [Z].`

### Step 4: Rewrite so it sounds natural
- `[Action] → [result] → [proof]`
- `[Result] by [action]`
- `[Accomplishment], measured by [Y], through [Z]`

### Step 5: Check if the point is portfolio-grade
- Is the accomplishment clear? Is the proof real? Is the method concrete?

## When To Use XYZ

Use XYZ when the point describes an `accomplishment with evidence`:
- performance improvements
- cost reductions
- revenue/growth outcomes
- reliability gains with measurable proof
- workflow efficiency gains
- ranked or selective achievements

## When XYZ Is NOT The Best Fit

- broad role summaries
- long-running responsibilities without one key outcomes
- product overview sentences
- technology scope statements

## Where To Use XYZ In This Portfolio

1. **Role Overview** — 1–2 strong XYZ points in a 4–5 point overview
2. **Workstream Cards** — 1–2 XYZ-shaped bullets per card
3. **Project Overview** — when there's a measurable improvement
4. **Deep Dive Sections** — whenever a subsystem has measurable payoff

## Point Patterns That Work Well

### Pattern A: Result → Proof → Method
`Improved [X] by [Y] through [Z].`

### Pattern B: Action → Result → Proof
`Built/Designed/Implemented [Z], which achieved [X] by [Y].`

### Pattern C: Accomplishment → Scale → How
`Accomplished [X] for [scale], by [Z].`

## Practical Rules For This Repo

1. Use XYZ when the result is already known and should lead the sentence.
2. Keep the final point to one strong sentence whenever possible.
3. Do not invent `Y`; if the metric is weak, use concrete proof instead.
4. Keep `Z` visible enough to show real work.
5. Avoid turning the point into a raw formula.
6. If the point needs more setup than result, use STAR instead.
7. Prefer one sharp metric over several weak ones.

## If There Is No Numeric Metric

Use proof types like: deterministic access control, easier auditing, faster debugging, reduced manual coordination, better portability, reusable orchestration logic, lower token waste, stronger observability.

## Reusable Compression Prompts

- `What exactly was accomplished here?`
- `How can I prove it?`
- `What number, scale, or concrete evidence exists?`
- `What did I actually do to create that outcome?`
- `Can this point start with impact without becoming vague?`

## Final Principle

XYZ is a way to make a point `earn its claim`. The best XYZ points feel like: visible accomplishment, believable proof, concrete method.
